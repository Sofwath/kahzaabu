# ADR 0008 — Quality evaluation methodology

**Status**: Accepted (2026-05-21)

## Context

Today kahzaabu has 78 unit tests that verify correctness — handler signatures, schema migrations, parser invariants, validator coercion. They prove the code does what we said it does. They prove **nothing** about whether the OUTPUTS are good. The curator may produce wrong contradictions, the decomposer may produce useless questions, the verifier may declare evidence in the wrong direction — and every existing test would still pass.

The literature handles this with held-out evaluation sets:

- **FEVER** (NAACL 2018) — 145k Wikipedia claims with labels; baseline accuracy 31.87%, current SOTA ~80%.
- **AVeriTeC** (EMNLP 2023) — 4,568 real-world claims with Q&A evidence; SOTA ~0.50 macro-F1.
- **RAGAR** (arXiv 2404.12065) — F1=0.85 on political fact-checking.

None of these are directly applicable: they're Wikipedia/news-corpus claims, not Maldives Presidency speech. But the methodology is portable.

## Decision

Build a held-out evaluation pipeline specific to kahzaabu's domain:

**Golden sets** under `tests/golden/`, hand-labeled by the human-in-the-loop:

```
tests/golden/
  extractor/           20-30 articles with expected claim lists
  decomposer/          20-30 claims with expected question sets
  matcher/             20-30 claim pairs labeled SAME / DIFFERENT
  contradictions/      20-30 claim pairs labeled with 4-way verdict
  verifier/            20-30 fact-checks with expected verdict_label
```

Each fixture is a small JSON file: `{input, expected, notes}`. Notes carry the reasoning so a future maintainer can re-judge the label if the data shifts.

**`kahzaabu eval` CLI** runs each pipeline stage against its golden set, producing:

- Precision / Recall / F1 per stage
- Confusion matrix per categorical-output stage
- Cost: total LLM spend for one eval run (capped at $5 to avoid runaway)
- A delta against the previous eval (stored under `data/eval_history.jsonl`)

**Prompt regression detection**: each LLM-call site embeds a `PROMPT_VERSION` constant (already pattern from `qna_agentic.py`). When `PROMPT_VERSION` changes, CI requires a new eval run; the test fails if F1 drops by more than 5% on any stage without an explicit ADR override comment.

**CI integration**: full eval is too slow + costly for every CI run. Instead, CI runs the SMALL eval (10 examples per stage, ~$0.20 per run) on every PR; the full eval runs nightly via `hermes cron`.

**Reporting**: `docs/EVAL_RESULTS.md` is auto-regenerated on every release, with the per-stage F1 numbers and a comparison row showing kahzaabu vs published benchmarks where applicable.

## Alternatives considered

- **Crowdsource the golden set via Prolific / Mechanical Turk.** Rejected — political claims about the Maldives Presidency require domain knowledge most general-purpose annotators don't have. The annotators must read Dhivehi, understand local context, and recognize names. Realistically: the project owner labels them, with a process to grow the set as collaborators join.
- **Use a held-out portion of the live corpus.** Rejected — the labels would be auto-generated by the same pipeline we're trying to evaluate, defeating the purpose.
- **Skip evals, ship without quality measurement.** Rejected — the explicit goal is a reference project. No reference project ships without F1 numbers.

## Consequences

**Positive.**

- Real comparability with FEVER / AVeriTeC / RAGAR. The README's "we cite these papers" claim acquires meaning.
- Prompt drift detection — a worse prompt blocks merge.
- A path for collaborators to grow the golden set without touching pipeline code.

**Negative.**

- Annotation work — the initial golden set is ~150 hand-labeled fixtures across 5 stages. Realistically ~6–8 hours of focused labeling for the project owner.
- Eval LLM cost: ~$0.20 per CI run × ~20 runs/month = $4/month overhead. Trivial.
- A regressed F1 on the eval set blocks merge — annoying when you're shipping something orthogonal to quality. We accept the friction; the override mechanism (ADR-justified bypass) handles edge cases.
